**测试日期:** 2026-04-07
**测试环境:** MacBook Pro (Apple Silicon), macOS

## 测试配置

- **工具版本:** v0.1.0
- **测试机器:** neo的MacBook Pro (ARM64)
- **Rayon 线程数:** 默认 (逻辑 CPU 核心数)

## 与 ripgrep 对比

| 指标 | Rust Search CLI | ripgrep | 比率 |
|------|----------------|---------|------|
| 总时间 | 3.54s | 1.63s | 2.17x slower |
| CPU 利用率 | 196% | 798% | - |
| 匹配行数 | 399,462 | 381,460 | 多 4.7% |
| 吞吐量 | 112K 行/秒 | 234K 行/秒 | 2.1x slower |

**分析:**
- ripgrep 利用了更多 CPU 核心 (798% vs 196%)，说明我们的并行化还有优化空间
- Rust Search CLI 匹配更多行，可能是因为没有完全跳过二进制文件
- 时间主要花在文件 I/O 和正则匹配上

## 性能瓶颈分析

1. **文件 I/O**: 每次搜索都 `read_to_string` 整个文件，大文件会有内存压力
2. **正则引擎**: `regex` crate 本身已经很快，但不是最高效的选择
3. **并行化**: Rayon 的工作分配策略可以进一步优化
4. **二进制检测**: 目前检测逻辑比较简单，可能漏掉一些二进制文件

## 优化建议

### 高优先级
- [ ] **内存映射文件 (mmap)**: 对大文件使用 mmap 减少内存复制
- [ ] **更好的工作分配**: 按文件大小分配任务，避免大文件拖慢整体
- [ ] **流式读取**: 使用 BufReader 替代 read_to_string

### 中优先级
- [ ] **正则引擎切换**: 考虑使用 `regex-lite` 或 `re2` 风格的引擎
- [ ] **贪婪过滤**: 在遍历阶段就排除不需要的文件
- [ ] **增量搜索**: 支持从上次位置继续搜索

### 低优先级
- [ ] **SIMD 加速**: 使用 SIMD 指令加速字符串搜索
- [ ] **缓存**: 对常用模式的结果做 LRU 缓存

## 结论

Rust Search CLI 在中小型项目上表现良好 (毫秒级响应)，适合日常开发使用。与 ripgrep 相比有 2x 左右的性能差距，主要原因:
1. ripgrep 使用了更激进的并行化策略
2. 文件 I/O 路径可以优化
3. 二进制文件过滤不够完善

**下一步目标:** 将性能差距缩小到 1.5x 以内。
